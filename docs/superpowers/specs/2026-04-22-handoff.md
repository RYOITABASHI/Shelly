### What the review agent found (commit 70e95340 rationale)

The four pre-session commits all assumed `handleFocusPane` in `PaneSlot`
ran on every pane tap. It does not: `ShellyTerminalView.dispatchTouchEvent`
(at `modules/terminal-view/.../ShellyTerminalView.kt:171-176`) calls
`parent?.requestDisallowInterceptTouchEvent(true)`, which prevents RN's
parent `<View onTouchStart=...>` from receiving the gesture. So body
taps never updated the 4 focus stores; only header/tab taps did.

Fix in `70e95340` — emit a new native event `onFocusRequested` from
`onSingleTapUp` carrying `sessionId`, subscribe in `TerminalPane` and
mirror the 4-store handoff. Files:
- `modules/terminal-view/android/.../ShellyTerminalView.kt`
- `modules/terminal-view/android/.../TerminalViewModule.kt`
- `modules/terminal-view/src/NativeTerminalView.tsx`
- `components/panes/TerminalPane.tsx`

### Speculative fix + diagnostics in 30dc9d02

`onSingleTapUp` was:
```kotlin
terminalView.requestFocus()
imm?.showSoftInput(terminalView, 0)
```

Replaced with:
```kotlin
terminalView.isFocusable = true
terminalView.isFocusableInTouchMode = true
val rootFocused = rootView?.findFocus()
if (rootFocused != null && rootFocused !== terminalView) rootFocused.clearFocus()
val reqOk = terminalView.requestFocusFromTouch()
// ...
val imeShown = imm?.showSoftInput(terminalView, InputMethodManager.SHOW_IMPLICIT) ?: false
Log.i(TAG, "onSingleTapUp sess=$sid reqFocus=$reqOk hasWin=$hasWinFocus focused=$isFocused ...")
```

Three compounded speculative changes:
1. Force `isFocusable + isFocusableInTouchMode` before requesting.
2. Explicit `clearFocus()` on whatever sibling currently owns focus.
3. `requestFocusFromTouch()` + `SHOW_IMPLICIT` flag.

Plus one `Log.i` per tap with requestFocus result, hasWindowFocus,
isFocused, focusable flags, visibility, width/height, showSoftInput
return, previous-focus-owner's class+hash, and current view's hash.

If the speculative fix works: bug closes, remove the noisy log in a
follow-up commit.
If it doesn't work: the log tells us which layer is refusing — focus
refusal, no window focus, IME returning false, or sibling still
holding focus after clearFocus.

### Follow-up diagnosis after user reproduced on latest APK

ADB wireless was connected to `SM_F956Q`
(`adb-RFCX71399SK-K0piCb._adb-tls-connect._tcp`). `scrcpy` was installed
via winget and launched as `Shelly-Device`.

Captured logcat showed the critical failure:

```text
Ignoring showSoftInput() as view=com.termux.view.TerminalView{3652cbf ...} is not served.
servedView=com.termux.view.TerminalView{2939ab8 ...}
onSingleTapUp sess=shelly-1 reqFocus=true hasWin=true focused=true ... imeShow=false
```

Interpretation: native focus moved to the tapped right-pane
`TerminalView`, but Android IME's served view stayed bound to the first

- `focusCommand()` now uses `requestFocusFromTouch()`.
- Both `focusCommand()` and `onSingleTapUp()` call
  `InputMethodManager.restartInput(terminalView)` before `showSoftInput()`.
- A next-frame retry calls `showSoftInput()` again and logs
  `onSingleTapUp.retry ... imeShow=...`.

- Every session tab now shows its stable hue even when inactive.
- Active tab gets stronger fill, border, and bottom accent line.

### Terminal wrapper grey flash

User observed slight grey tint/background when switching session tabs.
No code intentionally changes terminal background per session. Likely
cause: RN wrapper/container used theme `c.background`, which can differ
from the native terminal theme background and briefly show through during
tab rebind/repaint.

Fix in `eafb01b1`:

- `TerminalPane` outer container background now uses
  `terminalColorScheme.background`.
- `NativeTerminalView` wrapper style also uses
  `terminalColorScheme.background`.

### Duplicate session tab mounted in multiple panes

User reported layout breakage when opening the same terminal session tab
in multiple panes. Root cause is native architecture: one
`ShellyTerminalSession` owns one shared `TerminalSession.emulator`, size
state, PTY resize, and one `onScreenUpdateCallback`. Attaching that same
session to multiple `TerminalView`s makes them fight over:

- emulator rows/cols via `TerminalView.updateSize()`
- PTY resize via `ShellyTerminalSession.resize()`
- redraw callback assignment (`onScreenUpdateCallback` is overwritten)

Fix in `b61f2d2a`:

- If a user selects a session already visible in another terminal pane,
  do not mount it twice. Swap the two panes' `slot.sessionId` bindings
  instead.
- Closing a session rebinds every terminal pane that still references it
  to the first remaining session.

Verification target:

- Open 2-4 terminal panes.
- Try selecting the same session tab in pane 2 while pane 1 owns it.
- Expected: pane 1 and pane 2 swap sessions; no duplicate rendering, no
  layout corruption.

## Track 4 — Browser pane fixes

### YouTube/video stopping frequently

BrowserPane had a `visible` effect that injected JS to pause all playing
videos when hidden and play them again when shown. Because BrowserPane is
kept mounted in hidden mode after first opening, pane switches/type
changes could repeatedly call `video.pause()` / `video.play()`.

This is hostile to YouTube's player state: play promises can be rejected,
ads/player state can desync, and autoplay policies can treat resume as
not user-initiated.

Fix in `772310e8`:

- Removed the hidden-pane pause/resume JS injection.
- WebView remains mounted and page media state is left to the page.

### Browser pane density in 4-pane layouts

User reported 4-pane browser was too large/chrome-heavy, making text
input hard. The fixed-size URL bar, bookmark bar, and bottom input bar
consumed too much of a small pane.

Fix in `ebb69e98`:

- BrowserPane reads `MultiPaneContext` pane width/height.
- Compact mode when width `< 430` or height `< 380`:
  - URL toolbar padding/icon/input sizes shrink.
  - Bookmark bar becomes icon-only.
  - Bottom `PaneInputBar` is hidden.
  - WebView `textZoom` becomes `85`.
- Tiny mode when width `< 320` or height `< 300`:
  - Bookmark bar is hidden entirely.

Verification target:

- p4 layout with Browser + Terminal panes.
- Confirm browser chrome is compact and text input in neighboring panes
  remains practical.
- Confirm YouTube still plays while switching focus/panes.

## Track 2 — Termux-host claude-code resurrection

Claude-code 2.1.113+ ships as a Bun SEA binary with ELF interpreter
`/lib/ld-musl-aarch64.so.1` that Android bionic does not provide. The
npm-installed claude was hitting the postinstall stub that prints
"native binary not installed" and exiting 1.

### Actions taken (Termux host only, Shelly APK untouched)

1. Located the musl loader from prior bug #117 exploration:
   `/data/data/com.termux/files/usr/tmp/bun-sea-test/alpine/lib/ld-musl-aarch64.so.1`
2. Copied to stable path: `$PREFIX/share/shelly-loader/ld-musl-aarch64.so.1`
3. Replaced `$PREFIX/lib/node_modules/@anthropic-ai/claude-code/bin/claude.exe`
   with a wrapper that invokes `env -u LD_PRELOAD $LOADER $BIN "$@"`.
   (`LD_PRELOAD=libtermux-exec-ld-preload.so` is cleared because the
   bionic-targeted shim fails symbol resolution under musl.)
4. Edited `~/.bashrc` — removed the old `alias claude="TMPDIR=... claude-cli"`
   line (CJS-era artifact) and the related `claude-fix` alias.

Result: `claude --version` → `2.1.114 (Claude Code)` in a fresh bash.
The `cli.js not found` error was a stale shell alias, not a broken
install.

### What remains broken on Termux

- **claude /login and codex login**: OAuth browser flow cannot complete
  in Termux (no working xdg-open under musl for the SEA binary).
  Standing workaround: credentials transplant from a PC/browser device.
  See `2026-04-20-claude-credentials-transplant.md`.
- Options discussed but not yet chosen:
  - A. Use `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` — separate billing.
  - B. OAuth credentials transplant from another device — subscription
       passes through but refresh-token expiry forces re-transplant.
  - C. Manual OAuth URL copy/paste flow — to be tried for codex.

`gemini` is unaffected by the Bun SEA transition; still runs under node.
