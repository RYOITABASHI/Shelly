# Shelly v5.1.0 — Reliability pass + verified-latest CLIs

A reliability-first release. After two weeks of churn, we spent the
final four days running scripted on-device gauntlets on a Samsung
Galaxy Z Fold6 (Android 16, OneUI 6) instead of adding features. The
release focuses on the regressions we could reproduce in the
terminal / IME / paste path AND introduces a per-launch verified-latest
update path so users no longer wait up to 24 h before a fix lands on
their device.

The Claude Code Bash tool works on first launch on the default tier;
Gemini / Codex npm packages and the native Claude / Codex runtime path
now check upstream on every launch and promote any newer version that
passes the smoke gate. Default Claude remains pinned to 2.1.112 to
keep the cli.js Bash tool working — opt into the latest Bun SEA via
`SHELLY_PREFER_MUSL_CLAUDE=1`.

This is the recommended build for new installs.

## Highlights

### Claude Code Bash tool works out of the box (default tier)

Claude Code 2.1.112 (the last pure-JS `cli.js` release) is the **default
runtime**. The Android-specific subprocess fixes for
`child_process.spawn({ shell: true })` are wired in, so the **Bash tool,
MCP helper, and editor spawns actually run** on first launch — no proot,
no manual configuration.

The latest Claude Code (the Bun SEA build, 2.1.113+) is **opt-in** via
`export SHELLY_PREFER_MUSL_CLAUDE=1`. CI now compiles a separate
musl-targeted `libexec_wrapper_musl.so` (distinct from the bionic
`libexec_wrapper.so` that handles the default tier) so that `LD_PRELOAD`
can intercept `/bin/sh` and `/bin/bash` calls inside the SEA. This is
**expected** to fix `/bin/sh` for spawn paths that go through the musl
dynamic-symbol dispatch (the standard Bun child-process route on Linux).
Final release validation must confirm Claude Code's Bash tool on-device
on the SEA build — see the `PENDING` row in the verification table
below.

### Per-launch verified-latest auto-update (NEW in v5.1.0)

Earlier releases gated both updaters behind a 24 h time marker. Even
when upstream shipped a fix or new feature (codex 0.125.0 with GPT-5.5
model recognition is the canonical example), users could wait up to a
full day before the bg update tried to pull it. v5.1.0 replaces the
pure time gate with a **per-launch verified-latest check**:

- On every shell launch (rate-limited by a 1 h cooldown), Shelly fetches
  metadata only — `npm view <pkg> version` for the NPM tier, GitHub
  releases for the native Codex Termux fork, npm registry for the
  native Claude SEA — totalling ~30 KB and ~3-5 s in the background
- If any package's installed version differs from upstream **and** the
  upstream version is not in our `~/.shelly-cli/.failed-versions` /
  `~/.shelly-runtime/.failed-versions` cooldown list, the full updater
  fires immediately (not waiting for the 24 h marker)
- The full updater pipeline is unchanged — staging install, smoke gate,
  atomic promote, rollback on failure
- A version that fails its smoke gate is recorded in `.failed-versions`
  with a 1 h cooldown (override via `SHELLY_FAILED_VERSION_COOLDOWN`).
  The cooldown auto-recovers if upstream republishes the same tag with
  a fix
- Network failures and exceptions do **not** poison the cooldown list —
  only proven bad versions (failed smoke) are blocked
- The 24 h marker remains as a fallback for the offline-all-day
  pathology

Net result: a Codex release at 12:00 UTC reaches users within 1 h of
their next shell launch instead of up to 25 h. Bandwidth cost is one
metadata fetch per package per hour (in practice, per launch).

The two pre-existing tiers continue with their original responsibilities:

#### NPM-tier health check for the bundled JS CLIs

`__shelly_bg_cli_update` runs in the background and stages a fresh
install of:

- `@anthropic-ai/claude-code@2.1.112` (pinned — the last pure-JS release)
- `@google/gemini-cli@latest`
- `@openai/codex@latest`

into `~/.shelly-cli.staging/`, then runs a `--version` smoke test on
each CLI. **The tree promotes by same-filesystem renames only if all
three pass.** The current live tree is moved to `.prev` first, then the
staging tree is moved to live; if the second rename fails, `.prev` is
moved back. Each rename is atomic, but the transaction as a whole has a
brief no-live-tree window during the swap. Failure rolls back to the
last known-good. The bundled "golden" inside the APK is a third-tier
fallback — even a hosed update tree can't brick the CLI. This path is
sufficient for catching the kinds of breakage we have actually shipped
(missing native deps, sed-patch drift, post-install OOM).

#### Native runtime verified-latest for Claude / Codex

`shelly-runtime-update.js` is the native-runtime updater that fronts
the Claude Code Bun SEA path and the Codex Termux fork. It runs three
channels:

- **`verified`** (default) — try the newest npm release first; on smoke
  failure walk back through prior versions until one passes. Strictly
  better than a time-based cooldown — smoke is the safety mechanism, not
  the calendar.
- **`stable`** — only consider versions aged past
  `SHELLY_UPDATER_STABLE_DELAY_DAYS` (7 days by default). Walk back from
  there.
- **`latest`** — newest only, no walk-back. Fail loud if it doesn't
  smoke. Power-user / debug.

Smoke gates are layered:

- `--version` (no network, no auth) — always run
- `--print "reply OK"` for Claude (functional smoke that exercises
  DNS / TLS / auth / musl resolver / inference) when
  `SHELLY_UPDATER_FUNCTIONAL_CHECK=1` is set. Codex has no functional
  equivalent yet.

Gemini is **not** under this updater (gemini-cli is pure JS — it's
covered by the NPM-tier path above).

> **GPT-5.5 reality check**: Shelly can only ship what upstream
> publishes. As of the final pre-tag check, `@openai/codex@latest`
> reports `gpt-5.4 medium` (re-verify at tag time — see the pre-tag
> checklist). When OpenAI publishes a codex release that recognizes
> `gpt-5.5`, Shelly's verified-latest path picks it up the next time the
> smoke gate passes. There is nothing to enable in Shelly.

### Paste routing for CLIs and TUIs

Pastes used to be a maze of broken cases — Japanese leaking `[200~`
literals, two-line pastes submitting line-by-line into Claude, zsh
inside bash mishandling DECSET 2004, vim/less/nano receiving
bracketed-paste sequences they didn't ask for.

The new routing pipeline:

- All pastes funnel through a single `TerminalEmulator.paste()`
  chokepoint
- A marker file at `$HOME/.shelly_paste_force_tui` (auto-managed by the
  `claude` / `codex` / `gemini` bash wrappers via
  `__shelly_paste_tui_begin` / `__shelly_paste_tui_end`) **forces those
  CLIs onto the standard `\e[200~...\e[201~` bracketed-paste path**,
  even when bash is the parent shell. They are TUIs in bracketed-paste
  mode, not at a bash prompt.
- For an actual bash prompt with bracketed-paste enabled, paste is
  triggered with `\C-x\C-b` (0x18 0x02) instead of the raw `\e[200~`
  sequence, because Android's bionic bash readline strips the leading
  `ESC` byte. `\C-x\C-b` is bound to `bracketed-paste-begin` in the
  `emacs`, `vi-insert`, and `vi-command` keymaps in `.bashrc`.
- TUIs that haven't enabled DECSET 2004 (vim / less / nano) still get
  the `\r?\n → \r` per-line fallback so each line behaves as a discrete
  Enter.
- A `comm` check on the **session shell** (`/proc/<shellPid>/comm`)
  avoids the bash-only `\C-x\C-b` trigger when the session shell
  process itself is non-bash (e.g. user's `$SHELL` is zsh / fish);
  those sessions fall back to the standard bracketed-paste path. This
  does not detect arbitrary `zsh` / `fish` processes started as
  foreground children of bash — that remains a known limitation. The
  marker-file override covers only the Shelly-managed CLI wrappers
  (`claude`, `codex`, `gemini`) where the foreground child is a TUI
  rather than an interactive shell. Defaults to "is bash" on `/proc`
  read failure to preserve the current SSH / stream-session behaviour.

Net effect: paste a 50-line Japanese block into Claude — it shows up as
one message, not 50.

### Z Fold6 fold/unfold preset restore (bug #99)

Repro on the Z Fold6: open Shelly on the main display in single-pane,
fold to the cover screen, unfold back to the main display — the layout
auto-switches to a 3-pane preset (`p3l` / "1+2"), overwriting the
user's manual single-pane choice. The auto-switch effect was hard-coded
to apply `'1+2'` on every unfold transition.

The fix captures the user's actual preset BEFORE folding to single,
restores that preset on unfold, and persists the captured value to
AsyncStorage (`shelly:lastUnfoldedPreset`) so a process kill while
folded — common on Z Fold6 because OneUI aggressively kills foreground
apps when folded — does not lose the user's intent.

Five rounds of code review caught and closed three classes of edge
cases on the boot-folded → fast-unfold path:

- **Round 2 BLOCKER**: AsyncStorage hydrate is async vs `isFoldInner` is
  synchronous. Fix: re-fire the fold effect on hydrate completion via
  `useState(hydrated)` in the dep array.
- **Round 3 CRITICAL**: the round-2 re-fire alone did nothing — by the
  time hydrate resolved, `prev === curr` and the unfold branch was
  skipped. Fix: explicit corrective-restore branch that fires once when
  an unfold transition was deferred pre-hydrate.
- **Round 4 CRITICALs**: the corrective branch's early return forgot to
  update `prevFoldInnerRef`, causing double restore on subsequent width
  changes; AND the fold-branch AsyncStorage write was not gated on
  hydrate, allowing a fold-back-pre-hydrate to clobber the persisted
  preset with the constructor default. Both fixed in the shipped commit.

Round 5 walked six fold-cycle scenarios and verified ship-ready.

### Multi-pane keyboard rework

A single `TerminalImeHostView` now owns IMM's `mServedView` for the
whole Activity. Tapping between panes never requires IME migration on
Samsung OneUI — which fixes the stale-`mServedView` bug where typing or
Japanese composition could route to the wrong pane after a focus
switch. This was Test 3 in the release gate. PASS.

### Japanese rendering performance

`TerminalRenderer` now caches measured glyph widths for non-ASCII
codepoints in an 8192-entry LRU map. Previously every Japanese paste
called `Paint.measureText()` per codepoint per render frame, which on
a 200-character paste in multi-pane caused multi-second freezes and
forced task-kill-to-recover. The fix is expected to make the 200-char
2-up paste interactive again; final APK timing is tracked as PENDING
in the verification table below.

### Gemini CLI subprocess hardening

Three breakage classes are patched in CI before bundling:

- `shell: true` in the auto-update path (worked around Android bionic's
  empty `/bin/sh`)
- `shell: true` in the GrepTool preflight (same root cause)
- `execSync` in `commandExists()` which was silently failing every
  editor detection (vim / nano / cursor)

`!vim --version` and `!nano --version` from inside `gemini` now return
real output.

### Loud drift detection for bundled CLI patches

Both Claude Code's `shell:!0` and Gemini's `shell: true` /
`commandExists` patterns are now drift-checked twice:

1. **At CI build time** — if upstream renamed or restructured the
   minified bundle, the workflow fails loud with a descriptive error
   instead of producing a silently-broken APK.
2. **At on-device NPM-tier promotion** — `__shelly_bg_cli_update`'s
   staging tree is grepped for the same minified-JS patterns, blocking
   promotion if a future update shape-shifts the file we patch. (The
   native runtime updater for Claude / Codex does its own ELF / Bun-SEA
   shape detection — that's a different gate.)

We had silent breakage from this exact class of drift twice in v4.2.x.
The new checks are designed to fail closed instead of silently promoting
a known-bad shape.

## Security hardening

The LD_PRELOAD exec wrapper (`libexec_wrapper.so`) carries Codex audit
fixes (commit `38277f6b`):

- `dlsym(RTLD_NEXT, ...)` null-safety in every wrapper, surfacing
  `ENOSYS` instead of segfault when the dynamic linker hasn't completed
  libc wiring
- `SHELL` env validation against a trusted-prefix allowlist
  (`/data/user/0/dev.shelly.terminal/`, `/data/data/dev.shelly.terminal/`,
  `/system/bin/`) — blocks the `bash` lookup path-injection class
- `snprintf` truncation guard on `/bin/X` and `/usr/bin/X` synthesis,
  surfacing `ENAMETOOLONG` instead of silently truncating into a
  different executable name

Two HIGH-severity items from the same audit (PATH-resolved ELFs that
skip linker routing — bug #118; and a `is_elf` TOCTOU window — bug
#119) are tracked in `docs/superpowers/DEFERRED.md` as scheduled
post-release work — neither triggers on Shelly's current exec paths
(no PATH-only ELF spawn route is in use, and an attacker would already
need write access to the app-data directory to weaponize the TOCTOU).
We chose accuracy over hand-waving.

## Known limitations

Carried over from v4.2.0 / v5.0.0, restated for first-time users plus
items still tracked in `DEFERRED.md`:

- **APK is ~420 MB.** Tools (bash, node, python3, git, curl, ssh,
  sqlite3, rg, jq, tmux, vim, less, make) are bundled, not symlinked
  to Termux. If you already have Termux installed, Shelly does **not**
  use it.
- **Claude Code default is 2.1.112.** Latest Bun SEA Bash tool is
  opt-in via `SHELLY_PREFER_MUSL_CLAUDE=1` (see Highlights). Newer
  Anthropic features that depend on 2.1.113+ require this flag, and
  validation of the SEA Bash-tool path is PENDING (see Verification).
- **Keyboard avoidance is still imperfect on Android 15+ edge-to-edge
  layouts** for some keyboards / pane configurations. Tracked as bug
  #104 in `DEFERRED.md`.
- **Long-running sessions may still be killed by OEM battery
  management.** Foreground-service onboarding / Doze guidance
  improvements are tracked as bug #125. We have a foreground service
  today; the explainer flow is what's missing.
- **Paste into a remote SSH bash is line-by-line** when bracketed-paste
  is not enabled on the remote — that is documented bionic bash
  readline behaviour on the remote side, not a Shelly issue.
- **First-run OAuth flows for Claude Code and Gemini CLI** do not
  complete inside Shelly. Transplant credentials from a machine where
  auth works — see README "Bring your own credentials". Codex completes
  device-auth fully in-app.
- **GPT-5.5 is not available** because upstream `@openai/codex` has
  not shipped 5.5 model recognition yet (re-verified at the final
  pre-tag check). When it does, the verified-latest channel picks it
  up the next time the smoke gate passes.

## Upgrading

A clean install is recommended. Uninstall the previous Shelly first so
that:

- bash regenerates `.bashrc` from the new `BASHRC_VERSION`
- `~/.shelly-cli/` reinstalls against the new CI-patched bundle
- the marker-file root for paste routing is created fresh

```bash
adb uninstall dev.shelly.terminal
adb install -r shelly-v5.1.0.apk
```

If you keep your existing install: launch once and wait ~30 seconds for
`HomeInitializer` to detect the version bump and regenerate `.bashrc`.
The new `BASHRC_VERSION` will be visible in
`adb logcat -s HomeInitializer:D`.

## What's deliberately not in this release

In the spirit of shipping reliability over feature creep, several items
that came up in late-cycle code review (2026-04-25) are **explicitly
deferred** to v5.1.1 or later:

- **Shelly Doctor UI dashboard** (bug #122, P1, v5.1.1) — `shelly
  doctor` is already a CLI; the dashboard pane that surfaces auth
  status, CLI versions, runtime channel, storage / network state in
  one tap is next on the list.
- **HomeInitializer state-machine refactor** (bug #123, P2, v5.2.0) —
  the current 1500-line generator works; the refactor is pure
  technical debt cleanup.
- **Node compat preload shim** (bug #124, P2) — only useful if bug
  #117 Path A (the SEA Bash tool) regresses; we'll re-evaluate after a
  week of telemetry.
- **Foreground service onboarding UX** (bug #125, P3) — Doze exemption
  flow improvements, deferred until we have post-release feedback on
  actual failures.

We are not importing **any code** from the AnyClaw / OpenClaude /
Claw Code projects — their licensing / provenance and product-direction
tradeoffs do not fit this release. Shelly's solutions to Android exec
restrictions (`linker64` + `LD_PRELOAD` + musl trampoline) are
independent and documented.
