# Shelly 設計思想

## Shellyとは何か

Shellyは「ChatGPTのUI + 実際のターミナル実行」を1つのアプリに融合したもの。

- **チャットアプリ（ChatGPT等）** → 会話はできるが、実行できない。コードを生成しても、それをどこに置いてどう動かすかはユーザー任せ。
- **CLI（Claude Code等）** → 実行力は最強だが、ターミナルを使える人しか恩恵を受けられない。
- **Shelly** → チャットで話しかけるだけで、裏でCLIが実行する。ユーザーは「アプリ作って」と言うだけで、AIがファイル生成・git管理・テスト実行まで全部やる。その過程もチャットバブルで見える。

チャットアプリにはない実行力、CLIにはない手軽さ。その間を埋める唯一のポジション。

## ターゲットユーザー

ChatGPTやGemini、Claudeのチャットアプリで満足しているユーザーは、そもそもShellyを使わない。Shellyのユーザーは：

1. **CLIを使ってものづくりをしたいけれど、CLIを使いこなしきれない人たち**
2. **チャットベースで開発を進めたい人たち**
3. **ゴリゴリ開発もできるが、エンジニアではないから専門用語やプログラムに詳しくないという人たち**

つまり、チャットとターミナルをハイブリッドで扱えるツール。

## コアアーキテクチャ: 「チャットUIの裏でCLIが走る」

Shellyはチャットインターフェースだが、裏ではCLI（Claude Code, Gemini CLI, Codex等）が実際にコマンドを実行している。これがShellyの味噌。

### 4タブ構成

| タブ | 役割 | 対象 |
|------|------|------|
| **Projects** | プロジェクト一覧 + チャット履歴 | 全員 |
| **Chat** | メイン画面。右:ユーザー、左:AI。裏でCLI実行 | 全員 |
| **Terminal** | TTY生ターミナル。日本語入力可能 | 上級者 |
| **Settings** | 設定 + スニペット + Obsidian RAG + バックアップ | 初回設定時 |

旧設計は8タブ（Chat/TTY/Snippets/Creator/Browser/Obsidian/Search/Settings）だったが、これは「チャットアプリ」なのか「ターミナルアプリ」なのか分からない中途半端なUIだった。

新設計: **GPT/Claudeアプリに馴染みのあるユーザーが迷わないUI**。プロジェクト選ぶ → チャットする → 必要ならターミナル。

### ChatタブのUI

GPT/Claude/Geminiアプリと同じレイアウト:
- 右寄せ: ユーザーのメッセージ
- 左寄せ: AIの応答（アバター付き）
- コマンド実行結果: AIバブル内に埋め込み（折りたたみ対応）
- 長い出力: 自動折りたたみ、タップで展開
- 危険コマンド: 赤色の確認バブル

**思想**: 生ターミナルの内容をAIが要約・翻訳してチャット風に描画している。ユーザーが見るのはチャットだが、裏では本物のシェルが動いている。

### Projectsタブ

GPT/Claudeの左サイドパネルと同じ概念だが、スマホではサイドバーは操作が面倒（開く→選ぶ→閉じるの3アクション）なのでタブとして実装。

- チャット履歴（会話の部屋）
- プロジェクトフォルダ紐づけ（cwd）
- 新規チャット作成
- 検索（全履歴横断）

### Terminalタブ

Termux単体では**日本語入力すらできない**（GboardのIMEが正しく動かない）。ShellyのTerminalタブは、Shellyのキーボードレイヤー経由で日本語入力を可能にする。

これだけでもShellyの存在意義がある。

## Termuxless設計

全くの初学者が、ShellyはもちろんTermux自体も使ったことがない状態でインストールしても、Shellyの全機能を満足に使えるようにする。

### 5層アーキテクチャ

```
Chat UI（自然言語入力）
    ↓
Intent Router（ローカルLLMがタスクを分類）
    ↓
Tool Orchestrator（CLI管理・パーミッション翻訳）
    ↓
Environment Manager（ツール自動インストール・認証）
    ↓
Termux Bridge（コマンド実行）
```

### インテントルーティング

ユーザーは「何を使うべきか」すら知らなくていい。

例: 「ポートフォリオ作って」
1. ローカルLLMが意図を解析 → 「Claude Codeが最適」と判断
2. Claude Code未インストールを検知
3. チャットで案内: 「このプロジェクトはClaude Codeで開発するのが最適です。セットアップを開始しますか？」
4. ユーザーが承認 → 自動インストール → 認証フロー → そのまま開発開始

ツール選定もセットアップも、全部チャットの中で完結する。

## ペルソナ別のデフォルトエージェント

### ペルソナA: ローカルLLMを使うユーザー
- チャットの表に出てくるAIはローカルLLM
- 「こんにちは」→ ローカルLLMが応答
- 「アプリ作って」→ ローカルLLMが意図を解析し、Claude Code / Geminiに委譲
- ローカルLLMがインテントルーターとして機能する

### ペルソナB: ローカルLLMを使わないユーザー
- デフォルトエージェントは **Gemini CLI**
  - 無料枠がある（初学者がいきなり課金は厳しい）
  - Googleアカウントだけでセットアップ可能
  - 質問が自然言語で返ってくる

## モバイル開発が流行らなかった本当の理由

スペックの問題ではなかった。

### 1. 入力の問題
スマホのソフトキーボードでコードを書くのは辛い。だがShellyの設計思想で解決済み — 自然言語で指示するだけでCLIが書いてくれる。

### 2. ターミナルへのアクセス手段がなかった
Termuxはあったが、黒い画面にコマンドを打つだけで開発者以外には無理だった。日本語すら入力できない。Shellyがそのレイヤーを担う。

### 3. 「スマホで開発」という発想自体がなかった
ローカルLLMがスマホに実装できるサイズまで軽量化された今、モバイルでのVibe Codingは間違いなく一般化する。Shellyはその流れを先取りする。

## リソース消費の実態

| 処理 | メモリ消費 | 備考 |
|------|-----------|------|
| Shelly APK | ~40MB | UIのみ |
| Termux + Node Bridge | ~130MB | ほぼゼロ負荷 |
| + Gemini CLI | ~150-200MB | コマンド実行時のみ |
| + Claude Code CLI | ~200-300MB | コマンド実行時のみ |
| + llama-server (4B) | **~5GB** | 常時高負荷 |

ローカルLLMを除けば、全部合わせてもRAM 500MB程度。RAM 4GBの中スペック端末でもShellyは成立する。

## まとめ

Shelly = ChatGPTのUI + 実際のターミナル実行。

初心者はChat、上級者はTerminal。どちらも同じAIバックエンドと同じLinuxファイルシステムに繋がっている。スマホでの開発が流行らなかったのはスペックの壁ではなくUXの壁。Shellyはその壁を壊す。
